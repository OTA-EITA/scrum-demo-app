# スクラム講義カリキュラム

---

## 第1回：スクラムの基礎を学ぼう

### 目的
スクラム・アジャイルの概念を理解し、全体像を掴む

### アジェンダ

| # | 内容 | 時間目安 |
|---|------|----------|
| 1 | アジャイルって何？ | 10分 |
| 2 | スクラムって何？ | 10分 |
| 3 | アジャイル vs スクラム：違いを整理 | 10分 |
| 4 | Pros vs Cons：スクラムの良いところ・難しいところ | 10分 |
| 5 | スクラムでやること一覧 | 10分 |
| 6 | スクラムの流れ（スプリントサイクル） | 15分 |
| 7 | 実際に使うツールの紹介（Jira, Miro, Slack等） | 10分 |
| 8 | リファインメントって何？ | 10分 |
| 9 | スプリントレビューって何？ | 10分 |
| 10 | レトロは楽しいぞ！ | 5分 |

### ポイント
- 座学中心だが、質問を挟んでインタラクティブに
- 図解やイラストを多用して視覚的に
- 「現場ではこうだよ」という実体験を交える

## 第2回：スクラムを体験しよう

### 目的
実際にスクラムイベントを体験し、流れを体で覚える

### アジェンダ

| # | 内容 | 時間目安 | 形式 |
|---|------|----------|------|
| 1 | 前回の振り返り | 5分 | 全体 |
| 2 | スプリントレビューの流れ説明 | 10分 | 講義 |
| 3 | **デモアプリでスプリントレビュー体験** | 20分 | ハンズオン |
| | → アプリを触ってもらう | | |
| | → 改善点・感想を付箋に書き出す | | |
| | → 共有タイム | | |
| 4 | レビュー結果をバックログに落とし込む | 10分 | 全体 |
| 5 | リファインメントの流れ説明 | 10分 | 講義 |
| 6 | **リファインメント体験** | 25分 | グループワーク |
| | → 複数グループに分かれる | | |
| | → バックログアイテムを議論・見積もり | | |
| 7 | ファシリテーションのコツ | 10分 | 講義 |
| 8 | **レトロスペクティブ体験** | 15分 | グループワーク |
| | → KPT or Fun/Done/Learn形式 | | |
| | → 営業さんも各グループに参加 | | |
| 9 | クロージング・次回予告 | 5分 | 全体 |

### 準備物
- [ ] デモアプリ起動確認（Sprint Board）
- [ ] 付箋 or Miroボード
- [ ] グループ分け表
- [ ] タイマー

### ポイント
- **体験重視**：説明は最小限、手を動かす時間を多く
- グループワークで営業さんとエンジニアを混ぜる
- ファシリ緊張する話は共感を得やすい（講師も人間アピール）

## 第3回：改善サイクルを回そう

### 目的
スクラムの「継続的改善」を実感する

### アジェンダ

| # | 内容 | 時間目安 | 形式 |
|---|------|----------|------|
| 1 | 前回の振り返り | 5分 | 全体 |
| 2 | **改修版アプリでスプリントレビュー** | 25分 | ハンズオン |
| | → 前回の指摘がどう反映されたか確認 | | |
| | → 新たな改善点・感想を収集 | | |
| | → 「期待通りだったか？」の議論 | | |
| 3 | レビュー結果をバックログに追加 | 10分 | 全体 |
| 4 | **リファインメント体験（2回目）** | 25分 | グループワーク |
| | → 前回より慣れてきたか確認 | | |
| | → 見積もりの精度を意識 | | |
| 5 | **この講義自体のスプリントレビュー** | 15分 | 全体 |
| | → 講義の良かった点・改善点 | | |
| | → 今後の講義に向けたフィードバック | | |
| 6 | **この講義自体のリファインメント** | 10分 | 全体 |
| | → 次回以降やりたいことを整理 | | |
| 7 | レトロスペクティブ | 15分 | グループワーク |
| 8 | クロージング・全体まとめ | 5分 | 全体 |

### 第3回で見せる改修例（候補）
前回のレビューで出た意見から2〜3個選んで実装

| 改修内容 | 難易度 | インパクト |
|----------|--------|------------|
| 削除確認ダイアログ追加 | ★☆☆ | 見た目で分かりやすい |
| 戻るボタン（←）追加 | ★☆☆ | 要望が多そう |
| 優先度変更機能 | ★★☆ | 実用性が上がる |
| LocalStorage保存 | ★★☆ | 驚きがある |

### ポイント
- 「自分たちの意見が反映された」体験が重要
- 講義自体をスクラムで回すのは**メタ的で印象に残る**
- 最後は達成感を持って終われるように

## 全体を通してのTips

### ファシリテーション
- 沈黙を恐れない（5秒待つ）
- 最初に発言のハードルを下げる（「正解はないです」）
- タイムボックスは厳守（ダラダラ防止）

### グループ分け
- 4〜5人がベスト
- エンジニア + 営業混合で多様な視点を
- 毎回シャッフルすると交流が生まれる

### 使えるフレーズ
- 「現場ではこうなりがちですが...」
- 「私も最初は戸惑いました」
- 「正解はないので、チームで決めていきましょう」

## 必要なアプリケーション・ツール

### 必須

| ツール | 用途 | 備考 |
|--------|------|------|
| **GitHub** | ソースコード管理、Issues でバックログ管理 | 参加者は閲覧できればOK |
| **Miro** or **Figma** | 付箋ワーク、レトロ、ブレスト | Miro推奨（付箋機能が強い） |
| **Google Spreadsheet** | バックログ一覧、見積もり結果の記録 | 共同編集で使う |

### あると便利

| ツール | 用途 | 備考 |
|--------|------|------|
| **Slack** or **Teams** | 連絡、質問受付、ワーク中のチャット | 既存のものでOK |
| **Zoom** or **Google Meet** | リモート参加者がいる場合 | 画面共有必須 |
| **タイマーアプリ** | タイムボックス管理 | ブラウザのでもOK |
| **Jira** | 実際のスクラムツール紹介用 | デモで見せるだけでも可 |

### 講師側で準備

| 項目 | 詳細 |
|------|------|
| デモアプリ（Sprint Board） | `npm run dev` で起動、URL共有 |
| Miroボード | レトロ用テンプレート作成しておく |
| Googleスプレッドシート | バックログテンプレート用意 |
| スライド資料 | 第1回の座学パート用 |
